# A "CCP4i-style / file-oriented" mode in CCP4i2 — design note

> Working design note. Written to be **safe to commit**: no named individuals and no
> characterisations of people or user groups — only technical and product-design analysis.
> Still a living draft; refine freely.

## The question

Could CCP4i2 — via auto-generation and the *file-oriented* primitives it already owns
(`CMtzDataFile`, `CColumnGroup`/`CProgramColumnGroup`) — offer a mode and a set of tasks
in the style of the legacy file-oriented interface: one task = one program, files in,
files out, raw column (LABIN) selection, no imposed semantics on the reflection data?

Short answer: **technically yes, and more cheaply than first assumed — the architecture
already contains both data-model dialects.** The hard parts are conceptual and
product-strategic, not technical. This note separates those layers.

---

## Why this is feasible: the architecture is already bilingual

CCP4i2 already carries **both** reflection-data dialects as first-class citizens:

| | "Typed" (the CCP4i2 house style) | "File-oriented" (the legacy style) |
|---|---|---|
| File type | `CObsDataFile`, `CPhsDataFile`, `CMapCoeffsDataFile`, `CFreeRDataFile` | `CMtzDataFile` + `CMtzData.listOfColumns` |
| Column model | `contentFlag` + `CONTENT_SIGNATURE_LIST` enforce *which* columns | `CColumnGroup` discovers groups by pattern; `CProgramColumnGroup` maps program slots → real labels |
| Discipline | typed, validated, convertible (`as_FMEAN()` etc.) | none — "it's an MTZ, here are its columns" |
| Legacy analogue | (no analogue — this is the newer layer) | `_MTZ_file` + `_mtz_label_f` + a LABIN line |

The legacy task is a `.def` declaring `HKLIN _MTZ_file`, `FP _mtz_label_f`,
`SIGFP _mtz_label_sig` … plus a `.tcl` that reads the MTZ header to populate column
dropdowns. That maps **almost 1:1** onto `CMtzDataFile` + `CProgramColumnGroup` rendered
as a LABIN line. So a "file-oriented mode" is less *build new primitives* than **expose
primitives already present but kept behind the typed front door.** That reframing matters
for the cost analysis below.

### What's already there
- Registry is one dict entry per task (`core/tasks.py`); an `autogenerated` flag exists
  (currently *informational only* — nothing branches on it).
- Any task with no bespoke React interface auto-renders via `GenericInterface` +
  `COMPONENT_REGISTRY` widget lookup by CData class. Zero frontend per task.
- `CMtzDataFile` is already in the widget registry; `CColumnGroup` already has a widget.
- The `Import<Type>` family (`import_common.py`) proves a files-in/out task is ~6 lines of
  Python + ~25 lines of def.xml.
- `splitMtz`/`joinMtz`/`split_mtz_file` (gemmi-native) bridge file-oriented↔typed at will.

### The one genuine frontend gap
`CColumnGroupElement` today is a **display** chip (type + dataset + a selection toggle),
not yet the editable LABIN line ("pick a full MTZ, then assign its columns into named
program slots — FP, SIGFP, PHIB…"). The backend type (`CProgramColumnGroup`) exists; the
**editor widget is the real build.** It is one reusable widget, not per-task work — the
de-risking item, not the task corpus.

## Product-positioning context (neutral framing)

- **CCP4Cloud** — cloud-oriented, branched-project data model; pipeline-weighted usage
  (roughly 30:70 primitive:pipeline).
- **CCP4i2 (this branch)** — Qt-free desktop; flat collection of projects, jobs, files;
  positioned as a modern-toolchain, lower-maintenance line.
- **The legacy file-oriented interface (Tcl/Tk)** — remains in wide use. Its distinctive
  property is a near-pure "task = one primitive crystallographic action, files in/out"
  model, with no imposed mid/high-level data model. It carries an X11 dependency and a
  separate maintenance burden.

The strategic payload of a file-oriented mode is to let CCP4i2 absorb the file-oriented
usage pattern so the legacy interface can eventually be retired (removing X11 and a
separate codebase at the suite level).

---

## The case *for*

1. **It is the move that enables retiring the legacy interface.** That interface remains
   the only home of the pure files-in/out idiom; it can't be retired by being different
   from it, only by reproducing the idiom well enough that there is no reason to stay.
2. **The legacy corpus is machine-translatable.** ~181 `.def`+`.tcl` pairs; the `.def` is
   declarative and the program contract is an externalized template (see feasibility
   section). A scoped translation project, not an open-ended one.
3. **One atomic layer serves both humans and agents.** Fine-grained primitive tasks are
   what an automated/agentic driver wants — atomic, composable, documented operations
   rather than opinionated pipelines. The legacy task taxonomy is a ready-made ontology of
   primitive crystallographic actions; porting it yields a human file-oriented mode *and*
   an agent tool-library from one effort. Strongest forward-looking argument; lead external
   framing with it.
4. **Clean division of labour, fundable.** "CCP4i2 = the atomic/fine-grained layer for
   power users and agents; CCP4Cloud = the pipeline/branched-project layer." A file-oriented
   mode sharpens that positioning.

---

## The case *against*

1. **It is in tension with the low-maintenance positioning — unless framed carefully.**
   Adding a second UX idiom and a task corpus is, on its face, *more* surface. The
   reconciliation must be argued: (a) file-oriented tasks are auto-generated and share *all*
   execution/gleaning/DB infra — one engine, a different surface; (b) net suite-level
   maintenance *drops* because the legacy Tcl/Tk/X11 line goes away.
2. **It risks diluting the typed-data discipline.** Raw column picking re-opens the very
   ambiguities the typed model removes (column ambiguity, conversion errors, free-R
   confusion). Coherence requires a **bridge**, not two parallel worlds: file-oriented
   outputs must register in the project and a full `CMtzDataFile` must be splittable into
   typed mini-MTZs on demand (machinery exists — `splitMtz`). Decide deliberately whether
   the mode is *a different lens on the same data* or *a parallel data world*.
3. **A partial reproduction is worse than none.** An 80%-faithful mode invites the charge
   that it is a superficial re-skin. The bar is the full file-oriented experience (flat
   project/job view + LABIN line), not just a filtered task chooser.
4. **Positioning risk.** Reproducing the legacy interface can read as re-creating the past;
   the "atomic layer for humans + agents" framing is what keeps it forward-looking.
5. **Translation fidelity needs verification.** The `.def` translation is trivial and the
   program contract is largely declarative (the `templates/*.com` keyword templates). The
   genuine residue is (a) the small per-task logic in `<task>_run` and (b) arbitrary Tcl
   that leaks into `.com` guard expressions. Auto-translation yields a draft; each task
   needs an i2run test against known data — a mis-expanded `LABELLINE` is *silently wrong
   science*. Budget for verification, not just generation.

## The central objection: does this bake in a permanent schism?

A serious objection to the whole direction: by making the file-oriented usage pattern
comfortable inside CCP4i2, we may remove the forcing function that would move users onto
the modern model — legitimising the older pattern, fragmenting development effort, and
converting a *transitional* gap into a *permanent structural* one. This is a coherent
theory of how product lines accrete compatibility modes and should be taken on its merits,
not dismissed.

Relevant fact (currently **anecdotal**, load-bearing, must be verified): the legacy
file-oriented interface is reportedly still used by more people than either CCP4i2 or
CCP4Cloud. The modern lines have not yet drawn those users across — plausibly because
nothing offered so far has been compelling enough.

### Why this is hard to resolve: the non-adopters are not one population
Three distinct explanations are usually offered for why users have not moved. The key
point is that they imply **three different interventions** — and more than one can be true
of different sub-populations at once.

| Segment | Description | Right intervention | Bears on the schism risk? |
|---|---|---|---|
| **A — Settled preference** | Will not migrate regardless of what is offered | **No targeted investment**; addressed by natural turnover over time | **Yes** — building for this segment is the schism risk with no offsetting return |
| **B — Model-aversion** | Want access to CCP4 programs *without an imposed data model / project structure* | A genuinely **model-free surface** | **No** — *if* segment B is real and durable (see catch) |
| **C — Coverage gap** | Remain because a program they need is not yet wrapped | **Wrap the missing programs** — in the *normal modern idiom* | **Mostly no** — a file-oriented mode is the *wrong tool* for this |

Consequences:
- **Segment C needs no file-oriented mode.** It is won by adding coverage in the modern
  idiom — the cheapest, least controversial, highest-agreement intervention, and it bakes
  no schism.
- **Segment A should not be the target.** Investment there is precisely the schism risk.
- **Only segment B justifies a file-oriented mode** — and only if it is real and durable,
  not a re-description of segment A.

So the build-vs-don't decision collapses onto one question: **how large is segment B, and
is it durable?**

### Reframing "schism": codebase vs data-model
If segment B genuinely exists, the schism is *already present and durable* — a real
segmentation of needs, not a transitional failure. Serving it does not create it; it only
decides **who** serves it: a modern, maintained engine, or the unmaintained legacy line
with its X11 liability. The costliest schism is the **codebase** one — two stacks, two
maintenance burdens, X11. A file-oriented mode done well is **not a third pole**: it is a
thin surface on the **one** engine (shared execution / gleaning / DB). Collapsing the
legacy surface onto CCP4i2's engine **reduces** the codebase schism and is the only
credible route to retiring the legacy line.

### Where the objection retains force — and the condition on the whole idea
The reframe dissolves the *codebase* schism, not the *data-model* one. If segment-B users
live permanently in untyped full-MTZ data, they are excluded from the typed substrate that
makes the agentic/ML direction work. So the file-oriented direction carries a
non-negotiable condition: **the file-oriented↔typed bridge must be mandatory, not
optional.** File-oriented outputs must still glean and still be splittable into typed
mini-MTZs. That single property separates *"a model-free lens that still feeds the typed
world"* from *"a model-free enclave that fragments the future."*

### The synthesis worth naming
The technically-strongest answer — and the ideal agentic surface — is **atomic primitives
over a typed substrate**. The "file-oriented" experience becomes merely *whether the human
is shown the types*, not whether they exist. It serves both directions at once. It costs
more engineering than a pure reproduction and risks satisfying neither purist position,
but it is the only option that does not compromise the future.

### Resolution: convert the disagreement into a measurement
The decision hinges on unknowns that are **cheap to measure** and is currently settled by
competing anecdotes. Reversibility favours measuring first: *not* building is recoverable;
a hard-to-retract third pole is sticky.
- **Which programs do file-oriented users actually invoke?** Usage telemetry reads segment
  C *directly* and names the coverage gap.
- **Interviews / the user survey** separate A from B from C — the only honest way to size
  segment B.

Open question to settle explicitly: **is segment B durable or transitional?** Transitional
→ a file-oriented mode is a time-bounded retirement vehicle. Durable → there is a genuine
product-scope decision about whether the suite should serve a model-averse segment at all,
to be made in the open rather than implied by a feature.

## Data model: extraction vs reference, and the disk-truth invariant

"LABIN" is **not a widget question — it is a data-model question.** The widget is the
shadow; the object casting it is *how a reflection data object is represented*. There are
two representations, and "do we offer LABIN / a file-oriented mode" is really "do we admit
the second one."

### What the codebase actually persists today (grounded)

| Where | Carries column identity? |
|---|---|
| Django `File` model (`db/models.py`) | **No** — only `name, type, content (=contentFlag), sub_type, annotation, job_param_name, uuid` |
| Gleaner (`db/async_db_handler.py: glean_job_files`) | persists those fields only; **no column labels / dataset ids / groups** |
| On-disk params artefact | `baseName/relPath/contentFlag/subType/annotation` reliably; `fileContent.listOfColumns` **only incidentally** (if a job touched `fileContent`) — not an intentional, guaranteed record |
| Disk→DB reconstruction (`utils/reconstructDBFromXML.py`, `db/import_i2xml.py`) | **exists**, but rebuilds only `contentFlag/subType/annotation` — **column identity is not reconstructed** |

### The keystone: column identity is carried by *convention*, not by *data*
Object-at-column-level is recoverable solely as *(the file is a mini-MTZ) ∧ (contentFlag)
∧ (the canonical label table for that type)* — F,SIGF *because* it is an FMEAN. **There is
no persistent mechanism for non-canonical column labels, because the architecture is built
so it never needs one.**

### The original split/merge design did more than tidy bookkeeping
Physical extraction + canonical labelling is what makes the on-disk state **self-describing
with zero column-tracking metadata** — which is what makes the **disk-truth invariant hold
*for free* today**, and robust to exactly the failure that motivates it (a contaminated
SQLite on a distributed filesystem), because nothing load-bearing about column identity
lives in SQLite. That robustness is a real, currently-held property. The "monster MTZ +
references" idea is not blocked by *too late* so much as by this: **the moment non-canonical
labels are admitted, that free lunch is forfeit.** Arbitrary labels in a 47-column monster
cannot be recovered from convention; a reference object needs a durable, on-disk,
self-describing column-identity record or it violates the invariant the first time the DB
is lost. → "No persistent mechanism for non-canonical labels" is precisely *why* references
cannot be bolted on without new on-disk metadata. And the fix is **not better SQL** — a
column-aware Django model is only a rebuildable *index*; the truth must live in the on-disk
artefact.

### The design that follows: decouple logical object from physical encoding

| | **Logical data object** | **Physical encoding** |
|---|---|---|
| What | type + column roles + provenance (job, dataset) | mini-MTZ file **XOR** column-reference into a monster |
| Identity | durable, persisted on disk at column granularity | an implementation detail |
| Today | conflated with the file | only the mini-MTZ exists; identity = convention |

Once the logical object is persisted independently of its encoding, the two encodings
**interconvert**: a mini-MTZ trivially *is* a reference (file + all columns); a reference
*materialises* via `splitMtz`. **That interconversion is the "mandatory bridge" — and it is
load-bearing iff *both* encodings persist column identity to disk.** Today only the mini-MTZ
does (by convention). So **"make the bridge real" ≡ "build column-identity persistence."**

### How exposed are the wrappers? (migration blast radius — grounded)
The data-prep abstraction `makeHklin*`/`splitHklout*` (`core/CCP4PluginScript.py`) is the
insulation seam. A wrapper that routes reflection prep through it references only **container
attribute names + `contentFlag`**; the actual labels come from the file class's
`CONTENT_SIGNATURE_LIST`, never from wrapper code. Such a wrapper is representation-agnostic
— it works whether a data object is a physical mini-MTZ or a logical column-reference.

Audit of `wrappers/` + `pipelines/`:
- **Insulated (≈41 wrappers + ≈5 pipelines)** route through `makeHklin*`/`splitHklout*` —
  e.g. `refmac`, `buster`, `sheetbend`, `zanuda`, `i2Dimple`, `parrot`, `servalcat`. These
  need **no change** under a reference model.
- **Hand-rolled, contained (2 wrappers):** `freerflag` hard-codes `'FreeR_flag'`;
  `density_calculator` hard-codes `'FC'`/`'PHIC'` via direct `gemmi.add_column`. Both still
  emit through `splitHklout`, so output relabelling stays model-agnostic; only the literals
  need attention.
- **Hand-rolled, coupled (1 pipeline):** `crank2` constructs column lists from
  `GetLabel()` against intermediate files (~30 lines in `ccp4i2crank.py`) — couples to
  intermediate column labels; final output via `splitHklout` is safe.

**Implication:** the reference-model cost is **not spread across the corpus** — it is
concentrated in the persistence spine plus a finite exception list (~3 codebases, ~35 lines).
Make `makeHklin*`/`splitHklout*` reference-aware and the insulated set migrates for free.

### Lifecycle: a second blast radius (deletion and export)
There are **two distinct blast radii**, and they must not be conflated:
- **Migration blast radius (wrappers)** — *small* (previous subsection): the `makeHklin*`
  seam insulates ~41 wrappers; ~3 codebases hand-roll.
- **Runtime-lifecycle blast radius (deletion / export)** — *structurally harder under
  references*, and the more consequential of the two.

The extraction model holds five things coincident: **file ≡ data-object ≡ dependency-unit
≡ deletion-unit ≡ export-unit**. Job-deletion's "what breaks" closure and project/job
export's "what must I include" closure both *rely* on that coincidence — they reason at
file granularity (`FileUse` in/out edges, plus parent/child job lineage). The reference
model **unbundles** those five units, so every subsystem assuming they coincide must be
revisited — deletion and export most of all:
1. **Deletion.** When many logical objects share one physical monster, you cannot delete
   "the producer's file" — survivors may reference columns in it. Breakage becomes *partial*
   (column-granular), not all-or-nothing per file; "broken job" detection gets harder to
   compute and to explain.
2. **Export minimality.** Today an input *is* a minimal mini-MTZ. Under references the
   referenced file may be a 47-column monster of which a job used 2 columns. Minimal-closure
   export must then choose: **ship the whole monster** (simpler, bloated, may *leak unrelated
   columns* into an exported project) or **slice at export time** (materialise just the
   referenced columns — minimal, but new code that runs the very extraction references were
   meant to avoid).

**Consequence:** the dependency edge itself must become column-aware. It is not enough to
persist *output* column identity — input *references* must be persisted at column
granularity (which columns of which file a job consumed, as which object), i.e. **`FileUse`
becomes reference-aware**, so deletion blast-radius and export closure stay correct *and
reconstructable from disk* (the disk-truth invariant now covers dependency edges, not just
file metadata).

**Mitigating design decision — do references cross job boundaries?**
- **Constrained variant** (recommended first): each job writes its *own* file; references
  point only at whole produced files. Then file ≡ job still holds, deletion blast-radius is
  **essentially unchanged**, and the only residual new cost is export minimality
  (monster ⊇ referenced columns → ship-whole vs slice). Keeps the file-oriented UX win
  (non-canonical labels, point-at-columns, no forced typed split); forgoes cross-file dedup.
- **Shared variant**: references point into monsters produced by/accreted across other jobs.
  Buys storage/HKL dedup; costs column-granular deletion and the hard lifecycle cases.

Scope to the constrained variant first; relax to shared monsters only if storage dedup ever
proves worth the deletion/export complexity.

### Enabling work — bounded; type-ready, persistence-unready
The types already exist (`CMtzDataFile` = monster; `CColumnGroup`/`CProgramColumnGroup` =
selections). What is missing is *persistence and graph semantics*, not modelling:
1. **Intentional, mandatory** serialization of MTZ column identity into the on-disk params
   artefact (file, dataset, label→role, contentFlag, producing param) — not lazy/incidental.
2. **Reference-aware `FileUse`**: persist input references at column granularity, so the
   dependency graph is column-aware and reconstructable from disk.
3. A DB **index** (`FileColumn` model, or a JSON column-map on `File`) the gleaner
   populates — rebuildable, never authoritative.
4. **Extend `reconstructDBFromXML`** to rebuild that index *and* the reference-aware edges.
5. **Revisit deletion blast-radius and export closure** (`db/ccp4i2_django_dbapi.py`,
   `db/export_project.py`, `export_job.py`) for the unbundled file/object/deletion/export
   units — at minimum, decide ship-whole vs slice-at-export.

This is also a defensible engineering position on its own merits: it replaces a place where
CCP4i2 relies on convention with one where it relies on persisted, reconstructable data, and
strengthens the disk-truth invariant for every consumer (including, in principle, CCP4Cloud).
**Note the scope:** with lifecycle included, the reference model is a deliberate project, not
a side-effect of "let users pick columns" — a further reason to do near-term utilities as
mini-MTZ (extraction leaves deletion/export untouched) and schedule references separately.

### Project closure: external inputs/outputs and zombie jobs
The legacy file-oriented interface let start files live anywhere and let outputs be written
anywhere. In CCP4i2 the project hierarchy is a **closure**, and — importantly — this is
**structural, not policy**: a `File` is addressed by `(Directory ∈ {JOB_DIR, IMPORT_DIR},
name)`, and `File.path` is *computed* (`job.directory / name`, or
`project/CCP4_IMPORTED_FILES/name`). There is no field for an arbitrary path, so **a file
outside the hierarchy is not discouraged — it is unrepresentable** without a schema change.
`IMPORT_DIR` (`CCP4_IMPORTED_FILES`) is the only sanctioned membrane (what the Import tasks
use). Why the closure earns its keep:
- **External inputs** turn "broken" from a *graph property* (computable from the closure)
  into a *runtime filesystem property* (checkable only by touching disk — possibly a
  transient/distributed mount); they make export non-self-contained, and defeat
  reconstruct-from-disk (a project cannot describe what lies outside it). This is the
  zombie-job risk, made precise.
- **External outputs** are strictly worse: not gleaned, not first-class downstream, a
  dangling DB reference the moment anything external touches the file.

**Phenocopy the affordance without weakening the invariant:**
- **Inputs → import-on-use.** Let the user pick a file *anywhere*; at submission auto-copy
  it into `CCP4_IMPORTED_FILES` (`File(IMPORT_DIR, name)`) — exactly what the Import tasks do,
  made implicit. The "pick anything" feel, closure preserved, **zero schema change**.
- **Outputs → internal canonical + optional publish-out.** The tracked output always lands in
  `JOB_DIR` (gleaned, exportable); "also write to `/my/dir`" is an optional post-job copy-out
  the DB does not depend on. Precedent: `FileExport` already models a tracked copy-out and
  carries a `checksum`.

The only genuine case for true external-by-reference (no copy) is **large data where copy-in
is prohibitive** (unmerged / image data) — not the rump. If it ever lands, make it an
explicit, marked, **second-class** exception: a new `Directory` kind, a **mandatory content
checksum** with *detect-don't-prevent* semantics, and copy-at-export. A deliberate project,
out of scope for utilities.

## Automatic recapitulation: how translatable is the legacy corpus?

First impression of `~/Developer/ccp4-20260520/share/ccp4i/tasks/` is fair: data model,
procedural code, and UI are tightly interleaved in one file (the largest task `.tcl` is
~70 KB). But at the *file* level only. Two structural facts make it far more separable.

### Saving grace 1 — a proc-naming convention draws the layer boundaries
Each `tasks/<task>.tcl` decomposes into conventionally-named procs:
- `<task>_task_window` — **pure UI generation** (`CreateInputFileLine` ×511 across the
  corpus, `CreateLabinLine` ×362, `CreateOutputFileLine` ×211). **Discard** —
  `GenericInterface` replaces all of it.
- `<task>_run` — parameter massaging + validation + dynamic in/out file choice; returns
  0/1. Maps onto `validity()` / `processInputFiles`. (A typical one is ~70 lines of plain
  conditionals.) **The only genuinely procedural residue.**
- `<task>_setup` — defaults / typedefs → fold into def.xml qualifiers.

### Saving grace 2 (decisive) — the program contract is externalized declarative data
The program invocation is **not in the Tcl**. It lives in `templates/<task>.com` (162 of
them) — a small, regular templating DSL:

```
{ [IfSet $TITLE] } title $TITLE
$IFPATTERSON PATTERSON
$EXCLUDE_RESOLUTION resolution $EXCLUDE_RESOLUTION_MIN $EXCLUDE_RESOLUTION_MAX
1 labin
LABELLINE - $LABIN
```

Constructs: `$VAR` substitution, `{ ... }` / `$FLAG` guards, `IF/ENDIF`, `-`
line-continuation, `LABELLINE` for LABIN expansion. The contract is **already data, not code.**

### What each artifact becomes

| Legacy artifact | Becomes | Effort |
|---|---|---|
| `tasks/x.def` | `x.def.xml` | **mechanical** (deterministic script) |
| `templates/x.com` | keyword emission at run time | **one interpreter, then data** |
| `x_task_window` + UI helpers | nothing | **discard** |
| `x_run` + validation helpers | `validity()` / `processInputFiles` | **assisted, small** |
| the shared global `array` | the def.xml param set | **preserved as-is** (it *is* the contract) |

The last row is why the interleaving does not bite: UI and run communicate **only** through
the flat `array` keyed by `.def` parameter names — which *is* the def.xml parameter set. You
**amputate** the UI rather than untangle it; the contract survives the cut.

### The real leverage
Write **one interpreter for the `.com` DSL** and the bulk of each task stops being
code-to-translate and becomes data-to-run. For the simple tier a file-oriented task is then
almost pure import: def.xml from `.def`, command from `.com`, a thin `_run` port — much of
it *deterministic*. **This interpreter is the linchpin and the cheapest thing to prototype
first** (against `fft`/`cad`).

### Where it genuinely breaks
1. **Tcl leaks into `.com` guards** (`{ $USE_F2 && [IfSet $X] && ![StringSame $MODE PATTERSONI] }`).
   A deterministic interpreter covers ~90–95%; the tail needs per-task attention. Verify-heavy.
2. **Tier C is the interactive flagships** (the big refinement task with domains/TLS/etc.,
   and the pipelines). These do not auto-translate — and most already exist as bespoke
   wrappers in CCP4i2.
3. **Generation is cheap; trust is not.** Every auto-ported task needs an i2run test.

### Rough tiering of the ~181 tasks
- **Tier A (highly automatable, the majority):** single-program, declarative `.com`, thin
  `_run` (cad, fft, sfcheck, baverage, coordconv, pdbset, mapmask, mtz utilities).
- **Tier B (assisted):** real `_run` logic / multiple input modes / derived params.
- **Tier C (not automatic, a few dozen):** pipelines + interactive tasks — already largely
  built bespoke in CCP4i2.

### Bottom line
The corpus is more tractable than it looks, but in an inverted-value way: automation cheaply
delivers the *breadth* (the utility long tail that makes the chooser feel complete), while
the flagship daily-use tasks resist it (and are mostly already built). Realistic play:
**auto-translate Tier A/B for coverage + reuse existing bespoke wrappers for Tier C.**

## A concrete, de-risked path (if pursued)

**Phase 0 — Prove the chain on 3–4 representative tasks.** Exercise every hard edge: the
big refinement task (LABIN with many slots), `fft`/`cad` (MTZ column surgery), a scaling
task, a map task. Goals:
- Build the **editable LABIN widget** for `CProgramColumnGroup` (the one frontend gap).
- Confirm a file-oriented task is def.xml + thin wrapper + `GenericInterface`, no bespoke TSX.
- Confirm file-oriented outputs glean into the project and feed a downstream *typed* task
  (the bridge works end to end).
- Each ported task passes an i2run test against demo data.

#### Phase 0 *is* a showable artifact (full UX lifecycle in one capture)
Nearly every lifecycle beat already works for existing tasks — only the LABIN widget is new:

| Lifecycle beat | Status after Phase 0 |
|---|---|
| Task chooser → pick a file-oriented task | already works (register + categorise) |
| Task opens, renders interface | `GenericInterface` auto-renders from def.xml |
| Pick a **full** MTZ (`CMtzDataFile`) | existing file picker |
| **Assign columns into LABIN slots** | **the one new widget** |
| Set scalar params, hit Run | existing |
| Watch it execute, tail the log | existing |
| Outputs gleaned into project | existing |
| Open output / **feed into a downstream task** | existing — the bridge beat |

A single capture (chooser → LABIN assignment → run → result → output flows into the next
task) is in reach. **Caveat:** Phase 0 yields a file-oriented *task* inside the **normal
CCP4i2 shell**, not a legacy-skinned app (the flat project view + module taxonomy are
Phase 2). It proves *"a raw primitive task driven end-to-end with LABIN-style control,
output first-class"* — not yet *"the full legacy look and feel."*
- For the *atomic-layer-for-humans-and-agents* question → Phase 0 demonstrates it; record
  the bridge beat (file-oriented output → typed task) as the key demonstration.
- For the *does-it-feel-like-the-legacy-interface* question → needs a thin Phase-2 shell.

**Phase 1 — Auto-translation harness.** In dependency order: (1) the **`.com` DSL
interpreter** first — the linchpin, prototype against `fft`/`cad`; (2) `.def` → def.xml
generator (deterministic); (3) assisted `<task>_run` → `validity()`/`processInputFiles`
port. Discard `<task>_task_window`. Every output gated by an i2run test; `log()` whatever
fails to translate (especially `.com` guards with non-trivial Tcl). Target Tier A/B; leave
Tier C to existing bespoke wrappers.

**Phase 2 — File-oriented shell.** A mode toggle that (a) filters the task chooser to a
legacy-style module taxonomy (port `modules.def`'s folder structure) and (b) presents the
flat project/job/file view. The engine underneath is unchanged.

**Phase 3 — Retirement case.** Once coverage + fidelity are demonstrated, make the explicit
"retire the legacy interface / drop X11" case. That is the maintenance win that pays for the
effort.

---

## Recommendation

Pursue **Phase 0 only** first, as an experiment. It is cheap (one widget + a few tasks) and
answers the two decisive questions:
1. Does a faithful file-oriented task surface fall out of the existing engine with near-zero
   bespoke code? (Expected: yes.)
2. Does the file-oriented↔typed bridge hold, so the mode is *a lens* and not *a schism*?
   (If this fails cleanly, stop.)

In parallel, treat the larger question (whether to ship a durable file-oriented mode) as an
**empirical** one: size segments A/B/C via usage telemetry + survey before committing to a
permanent surface. The reference data model, if pursued, is gated on the column-identity
persistence spine (Data Model section).

### Q1 — Rump/utility programs: a *gated* fork, not just "build wrappers"
- **As mini-MTZ (extraction):** disk-truthful for free, ships fast, needs no new
  persistence — but re-imposes split/merge on model-averse users. *Coverage only* (a
  segment-C win); advances neither the data model nor segment B.
- **As reference (monster + non-canonical labels):** matches the file-oriented mental model
  and advances toward the gemmi-native, agent-friendly logical-object design — but is gated
  on the column-identity persistence spine, or it breaks the disk-truth invariant.

**Recommendation:** the reference model is the right long-term answer and utility programs
are the right pilot — **but only commit to the reference variant alongside the persistence
spine in the same piece of work.** Otherwise do utilities as mini-MTZ (coverage-only) and
schedule references as their own project. **Do NOT** ship references without on-disk column
persistence — the one option strictly *worse* than today. Target *pre-model* MTZ utilities
first (`cad`, `mtzutils`, `reindex`, sfcheck-likes), where non-canonical labels naturally
arise. Note the small blast radius: `makeHklin*`/`splitHklout*` insulates ~41 wrappers; the
exception list is ~3 codebases (`freerflag`, `density_calculator`, `crank2`).

**Keep the project closure either way.** Do not allow external start/output files — they are
unrepresentable without a schema change and reintroduce zombie jobs (see Project Closure).
Phenocopy the legacy "anywhere" feel via **import-on-use** for inputs and **internal output +
optional publish-out**. Zero schema change; no zombie risk.

### Q2 — NLP: model-agnostic; cloud-model-for-demo, never cloud-model-*required*
Neither "assume a specific cloud model" nor "minimal local NLP only" as architecture. A
hard dependency on a commercial cloud API is a governance / offline / cost / reproducibility
non-starter for a global academic suite; a tiny local model cannot do reliable
intent/parameter extraction. So:
- **Provider abstraction:** NL→task-graph behind a pluggable interface (cloud / local /
  none), degrading to *guided structured input* when no model is present.
- **The model-independent deliverable is the atomic-typed-task-as-callable-tool layer** —
  useful under every NLP outcome, and the same substrate the typed-primitive direction
  already builds. Scope the *tool layer*, not a parser.
- Separate **demo ambition** (drive with the most capable model) from **architecture
  commitment** (no model ever required). Governance note: NL prompts are usually *metadata*,
  not diffraction data — smaller privacy surface than it looks, but still opt-in.

### Q3 — Refinement demo: a *primitive slice*, paired with an auto-translated utility
- **Primitive refinement only**: `HKLIN`+`XYZIN` → `XYZOUT`+map coeffs, LABIN-driven. **Not**
  the full interactive task (domains/TLS/etc. — Tier C, days of work, not the point). State
  explicitly: a deliberate primitive slice / mechanism proof, not a parity claim.
- **Show the bridge beat in the same demo**: output flowing into a typed downstream task.
- **Pair with one auto-translated utility** (`.com`-interpreter prototype on `cad`): two
  small artifacts, two true stories — *"the file-oriented experience works"* and *"the long
  tail comes cheaply"*. The pairing avoids overselling auto-translation (the refinement
  slice would be hand-built).
